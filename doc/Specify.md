## 2. Governing Philosophy

This project is governed by four principles, listed in order of precedence. Every design decision, every line of code, and every architectural choice must be traceable to at least one of these principles.

**Principle 1 — Functional correctness over aesthetic complexity.**  
The system must do exactly what it declares it does. Cleverness that does not serve correctness is waste. No pattern, library, or abstraction is introduced unless it resolves a verified problem.

**Principle 2 — Scientific rigour and zero ambiguity.**  
Every requirement, every acceptance criterion, and every test case is stated without ambiguity. If a sentence in this document can be interpreted in two ways, it is wrong and must be rewritten. Precision is not pedantry; it is the precondition for correctness.

**Principle 3 — Universal over specific.**  
No value that is environment-specific, deployment-specific, or operator-specific may be hardcoded in source code. All such values are configuration. The system must be deployable in any conforming environment by changing configuration alone, with no modification to source code.

**Principle 4 — Replicability is a first-class deliverable.**  
Any team member must be able to reproduce any state of this system — local, staging, or production — by executing documented scripts. A feature that cannot be reliably reproduced by a third party from a clean checkout is not complete.

### What this philosophy explicitly prohibits

- Hardcoded credentials, hostnames, port numbers, or environment-specific paths in any source file.
- Mock objects or stub implementations in integration tests. Integration tests exercise real subsystems or synthetic data generated by deterministic code (see Section 8).
- Decorative code: unused imports, commented-out blocks, TODO comments not backed by a ticket.
- Documentation that describes what code does rather than why a non-obvious decision was made.
- Emoji, icons, or ASCII-art decoration in any project document or output.

## 3. Formal Project Goal

> **To provision, configure, and validate a production-grade MCP gateway on Google Cloud GKE using IBM ContextForge, such that AI agents belonging to the Baptist Health platform can discover, authenticate against, and invoke tools registered in the gateway through a single, governed access point, with behaviour that is verifiable by automated tests at every layer.**

This goal decomposes into three non-negotiable outcomes:

1. **Infrastructure outcome.** A GKE cluster exists in the GCP sandbox with ContextForge, Cloud SQL (PostgreSQL), and Cloud Memorystore (Redis) deployed and interconnected, reachable from within the GCP project network.

2. **Functional outcome.** The gateway correctly proxies requests to at least two classes of MCP target: (a) a vendor-hosted remote MCP requiring OAuth 2.0 authentication (GitLab), and (b) a virtual server created natively inside ContextForge.

3. **Operational outcome.** The deployment is reproducible from a clean state using a single entry-point script. Deployment steps, configuration parameters, and known failure modes are documented with enough detail that a team member who was not present during initial setup can reproduce the environment without assistance.

## 6. Definitions

All terms below are used with these precise meanings throughout this project. Any deviation from these definitions in any document or code comment is an error.

**MCP (Model Context Protocol).** The open protocol, defined by Anthropic, that standardises how AI agents discover and invoke tools. An MCP server exposes tools, resources, and prompts over a transport (SSE, Streamable HTTP, or stdio).

**MCP Farm.** The name of this project. Refers specifically to the governed infrastructure layer (ContextForge + GKE) that acts as a centralised registry and proxy for multiple MCP servers.

**IBM ContextForge.** The enterprise MCP gateway product deployed as the core of the MCP Farm. It has two logical subsystems: the AI Gateway (request routing and proxying) and the Configuration Registry (MCP catalogue, tool definitions, policies, virtual servers). Both subsystems require a PostgreSQL-compatible database and a Redis-compatible cache.

**Virtual Server.** A ContextForge construct that bundles one or more tool definitions into a single MCP-accessible endpoint. The tools inside a virtual server do not need to be independently MCP-compliant; ContextForge wraps them.

**MCP Proxy.** A ContextForge configuration that routes incoming tool invocations to an existing MCP server hosted externally (e.g., GitLab's hosted MCP endpoint). ContextForge handles authentication on behalf of the calling agent.

**Container Port Exposure.** The specific ContextForge configuration flag that must be set to `true` for the AI Gateway to bind to and expose its container port. This flag was the root cause of the failed Cloud Run deployment documented in the daily meeting transcript of 2026-05-20. This flag must be set explicitly in every deployment target.

**GKE (Google Kubernetes Engine).** The managed Kubernetes service on GCP used as the deployment platform for Phase 1.

**Cloud SQL.** GCP's managed relational database service. Used in this project with PostgreSQL engine as the persistent store for ContextForge's configuration registry.

**Memorystore.** GCP's managed Redis service. Used as the cache layer for ContextForge.

**OAuth 2.0 (PKCE).** The authorisation protocol used by vendor-hosted MCPs (GitLab, Snowflake). ContextForge must be configured to handle the OAuth 2.0 token exchange on behalf of agents.

**Synthetic data.** For any test that requires data (tool definitions, MCP registration payloads, configuration records), data that does not exist in a real system is generated programmatically using the `faker` Python library with a fixed random seed. Fixed seeds are mandatory for deterministic test replication. No hardcoded string literals are used as test data.

**Clean checkout.** The state of the repository after a fresh `git clone` with no pre-existing environment variables, credentials files, or installed packages beyond the operating system base. The replicability standard requires the system to be deployable from this state by following documented steps only.

**Cluster (SDD context).** A group of related implementation tasks that can be completed and committed as a logical unit. A cluster ends with a commit prompt. Clusters are the unit of progress in this project.

## 8. Data Standards

### 8.1 No mock objects in integration tests

Unit tests may use dependency injection. Integration tests must not. An integration test that invokes the ContextForge API must call a running ContextForge instance — local Docker or GKE — not a simulated one. A test that passes against a mock but fails against the real system is not a test; it is a liability.

### 8.2 Synthetic data generation

When a test requires data that does not come from a real external system (e.g., a test MCP server registration payload, a user record, a tool definition), that data is generated by a Python function using `faker` with an explicit, documented seed.

```python
from faker import Faker

SEED = 42  # Fixed. Must not change between runs.
fake = Faker()
fake.seed_instance(SEED)
```

The seed value `42` is the project default. Any test module that requires a different seed must document the reason. The seed must never be randomised or time-derived.

### 8.3 No hardcoded values

The following categories of value are always read from configuration, never hardcoded:

- Hostnames, IP addresses, and ports
- Credentials (passwords, tokens, API keys, certificate paths)
- Kubernetes namespace names
- GCP project identifiers, region, and zone
- ContextForge version tags
- Database names and connection strings
- Any timeout or retry parameter

Configuration sources, in order of precedence: environment variables, `.env` file (never committed), configuration file (`config/` directory, committed with placeholder values), default values in code (for non-sensitive parameters only).

## 9. Architecture Overview

The following describes the target state. This is not a prediction; it is a specification. Implementation must converge to this state.

```
                        [ AI Agent ]
                             |
                    (MCP tool invocation)
                             |
                    +------------------+
                    |  ContextForge    |
                    |  AI Gateway      |  <-- Container port exposure flag: TRUE
                    |  (port 8080)     |
                    +--------+---------+
                             |
              +--------------+--------------+
              |                             |
    +---------+----------+       +----------+---------+
    | ContextForge        |       |  MCP Proxy Target  |
    | Config Registry     |       |  (e.g. GitLab MCP) |
    | (port 8081)         |       |  OAuth 2.0 flow     |
    +---------+----------+       +--------------------+
              |
    +---------+-----------+
    |                     |
    v                     v
[ Cloud SQL          [ Memorystore
  PostgreSQL ]         Redis ]
```

**Deployment topology (GKE):**

```
GKE Cluster: mcp-farm-sandbox
  Namespace: mcp-farm
    Deployment: contextforge
      Container: contextforge (icr.io/...)
        Port 8080: AI Gateway
        Port 8081: Config Registry
    Service: contextforge-svc (ClusterIP)
    Ingress: contextforge-ingress (LoadBalancer or GKE Ingress)
    Secret: contextforge-db-credentials
    Secret: contextforge-redis-credentials
    ConfigMap: contextforge-config

GCP Managed Services:
  Cloud SQL: mcp-farm-pg (PostgreSQL 15)
  Memorystore: mcp-farm-redis (Redis 7)
```

**Local development topology (Docker Compose):**

```
docker-compose.yml
  service: contextforge  (same image, same config flags)
  service: postgres      (postgres:15)
  service: redis         (redis:7)
```

Local and GKE configurations must be structurally identical. The only permitted differences are connection strings and secret injection methods.

## 11. Testing Mandate

### 11.1 Test hierarchy

Every cluster must include tests at the level appropriate to what the cluster delivers. The absence of tests is a blocking acceptance criterion failure.

| Test type | Scope | Tool | When required |
|---|---|---|---|
| Unit | Single function or class | `pytest` | Any pure logic introduced |
| Contract | API request/response shape | `pytest` + `pydantic` | Any API integration |
| Integration | Two or more real subsystems | `pytest` | Every cluster from C1 onward |
| Progressive integration | Full system end-to-end | `pytest` (accumulated suite) | At the end of each cluster, the full test suite from all prior clusters runs and must pass |

### 11.2 Test naming convention

Test files are named `test_<cluster_id>_<subject>.py`. Example: `test_c1_local_environment.py`. This makes it unambiguous which cluster a test belongs to and allows selective execution by cluster.

### 11.3 Test execution

A single command must execute the full test suite:

```bash
make test
```

A single command must execute tests for a specific cluster:

```bash
make test CLUSTER=c1
```

Both commands must exit with code `0` on success and a non-zero code on any failure.

### 11.4 Test report format

All test runs produce a JUnit XML report at `reports/test-results.xml`. This file is never committed (it is in `.gitignore`). It is generated on every run for CI consumption.

## 12. Replicability Mandate

The following scripts must exist and be maintained throughout the project. They live in the `scripts/` directory. All scripts accept `--help` and document their parameters.

| Script | Purpose |
|---|---|
| `scripts/install.sh` | Install all system-level and Python dependencies from a clean state |
| `scripts/local_up.sh` | Start the local Docker Compose environment |
| `scripts/local_down.sh` | Stop and clean the local Docker Compose environment |
| `scripts/gke_provision.sh` | Provision the GKE cluster and managed GCP services |
| `scripts/gke_deploy.sh` | Deploy Kubernetes manifests to the existing GKE cluster |
| `scripts/gke_teardown.sh` | Destroy all GKE and managed GCP resources cleanly |
| `scripts/run_tests.sh` | Execute the full test suite (wraps `make test`) |
| `scripts/generate_synthetic_data.py` | Generate all synthetic data sets used by tests |

Every script is idempotent. A script that has already been run produces either the same result or a graceful no-op — never an error.

## 14. Commit and Branch Strategy

### 14.1 Branch model

All work for this project is done on the `main` branch during Phase 1, as this is a single-developer project in an early research phase. If a second developer joins, a feature-branch model is adopted.

### 14.2 Commit granularity

Each commit corresponds to one completed cluster (or a clearly named sub-unit of a cluster if a cluster is large). Commits are never made mid-cluster unless the work is independently verifiable.

### 14.3 Commit message format

```
<type>(<cluster>): <imperative summary under 72 characters>

<optional body: the why, not the what>

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

Type values: `feat`, `fix`, `infra`, `test`, `docs`, `refactor`, `chore`.

Examples:
```
feat(c1): add Docker Compose stack for local ContextForge environment
infra(c3): provision GKE cluster and managed Cloud SQL instance
test(c2): add integration tests for ContextForge gateway health endpoint
docs(c6): add deployment runbook with GKE provisioning steps
```

### 14.4 Commit prompt

At the end of each cluster, this document (via the implementing agent) will present a commit prompt in the following format:

```
COMMIT READY: Cluster <N> — <Name>

Suggested commit message:
  <type>(<cluster-id>): <summary>

Files to stage:
  <list of files>

Command:
  git add <files> && git commit -m "<message>"
```

The developer reviews, adjusts if needed, and executes the commit before work on the next cluster begins.

## 16. Acceptance Criteria — Project Level

The following criteria define the completion of Phase 1. All must be satisfied before the Jira ticket is closed.

1. A GKE cluster exists in the GCP sandbox and is reachable via `kubectl` using credentials documented in the runbook.
2. `kubectl get pods -n mcp-farm` returns all ContextForge pods in `Running` state.
3. The ContextForge AI Gateway health endpoint responds with HTTP 200 when called from within the cluster network.
4. The ContextForge Config Registry health endpoint responds with HTTP 200 when called from within the cluster network.
5. A `GET /v1/tools` (or equivalent ContextForge tools listing endpoint) returns a non-empty JSON array containing at least the GitLab MCP tools.
6. The full test suite (`make test`) passes with exit code 0 from a clean checkout against a configured environment.
7. `scripts/install.sh` followed by `scripts/gke_deploy.sh` reproduces the deployment without error on a second independent execution.
8. The deployment runbook (`doc/ADR/` or a dedicated `doc/Runbook.md`) contains all steps, commands, and expected outputs for a complete reproduction.
9. All open questions (Q1–Q6) are resolved and documented.
10. No secret, credential, or environment-specific value is present in any committed file.
